# AquaFantasia v2.0.22

고퀄리티 SD 해양 판타지 RPG 방향으로 전환 중인 PixiJS 기반 모바일 세로형 웹게임입니다. 현재 목표는 메뉴형 낚시 게임에서 벗어나, 플레이어가 직접 루미나 베이 마을을 돌아다니고 NPC, 건물, 항구, 건설 시스템을 자연스럽게 이용하는 첫 마을 RPG 기반을 완성하는 것입니다.

## 현재 방향

- 장르: SD 해양 판타지 RPG
- 화면: 모바일 세로모드 1080 x 1920 기준
- 렌더링: PixiJS 8 기반 2.5D 마을 월드
- 조작: 좌측 가상 조이스틱, 터치 이동, 우측 간소 메뉴, 캐릭터 시점 확대/축소
- 배포: GitHub Pages 가능 구조
- 저장: 로컬 저장 우선, Firebase 연동 준비 구조 유지

## v2.0.22 변경사항

- 낚시 화면의 `낚시 시작` 버튼 폭과 높이를 줄이고 pulse 애니메이션을 약하게 조정해 릴링 UI와 겹치지 않도록 보정했습니다.
- 낚시 화면의 우측 하단 메뉴 도크를 홈 화면과 같은 `[마을] / [가방][퀘스트][지도]` 구조로 다시 표시하도록 복구했습니다.
- 릴 감기 패널과 세로/가로 게이지를 화면 안쪽으로 강제 고정하고, 작은 화면에서 왼쪽 밖으로 밀리지 않도록 폭/좌표/그리드 최소값을 재조정했습니다.
- 기본 알림 팝업을 HUD 아래의 작은 알림 카드처럼 보이도록 위치와 크기를 보정했습니다.
- 홈 HUD의 캐릭터 정보 칩 검은 음영을 제거하고 밝은 아쿠아 톤으로 정리했습니다.
- 홈 화면의 `+ / - / 원점 / 건설 / 출항` 컨트롤 바를 HUD 오른쪽에 붙도록 재배치하고, 겹침 방지를 위해 HUD 우측 여백을 소폭 늘렸습니다.
- 건설 팝업을 더 크게 열고 카드/버튼을 읽기 좋게 키웠으며, 모바일 세로 드래그 스크롤을 유지했습니다.
- NPC 이동 AI에 예약 타일/목표 분산/짧은 양보 대기 로직을 추가해 주민들이 같은 동선에 겹쳐 서는 빈도를 줄였습니다.
- 기존 v3d_underwater UI/FX 에셋을 낚시 버튼, 건설 팝업, 알림 카드에 추가로 적용했습니다.
- `package.json`, `package-lock.json`, `APP_VERSION`, service worker cache, offline badge를 `2.0.22`로 동기화했습니다.
- README.md만 수정했으며 별도 `*_NOTES.md` 파일은 만들지 않았습니다.

## v2.0.19 변경사항

- npm registry/설치 실패 원인 진단 보강
  - 작업 컨테이너에서 `registry.npmjs.org` DNS 조회가 `EAI_AGAIN`으로 실패하면 패키지 코드 문제가 아니라 외부 npm registry 네트워크/DNS 접근 제한으로 판단
  - GitHub Actions에서는 public npm registry 강제, package-lock 내부 registry 오염 검사, 설치 로그 금지 문자열 검사를 계속 유지
  - `ci:registry:check` 스크립트와 workflow 진단 단계를 추가해, 실제 CI에서 네트워크 문제인지 lockfile 오염인지 구분하기 쉽게 함

- 낚시터 찌/게이지 안정화
  - Pixi 낚시 화면에서 캐스팅 전 찌가 미리 떠 있는 문제를 막기 위해 `idle` 상태에서는 bobber를 숨기고 캐스팅 시작 때만 표시
  - HTML fallback 낚시 화면도 `idle/success/fail` 상태에서는 찌를 숨기고 `casting/waiting/bite/reeling` 상태에서만 보이도록 phase 클래스를 추가
  - 낚시 게이지 패널을 화면 하단 안전 영역 안으로 고정하고, 작은 화면에서도 horizontal/vertical gauge, hold button, progress bar가 밖으로 밀리지 않도록 크기 제한
  - 릴링 중에는 최근 포획 패널을 자동으로 숨겨 게이지와 겹치지 않게 함
  - fallback 릴링 루프의 `const safe` 중복 선언을 제거해 TypeScript 빌드 실패 후보를 수정

- 낚시터 에셋 추가 적용
  - 기존 보유 에셋 중 물결 링, 해안 거품, 물고기 그림자, 작은 물튀김을 낚시 stage에 배치
  - 추가 낚시 ambient 에셋을 service worker precache에 연결해 오프라인 캐시 누락을 줄임

- 버전과 검증 갱신
  - `package.json`, `package-lock.json`, `APP_VERSION`, service worker cache, offline badge를 `2.0.19`로 통일
  - `npm run validate`는 v2.0.19 낚시터 안정화, registry 진단, README-only 규칙, 8방향 반전 방지를 함께 확인
  - 루트 문서는 계속 `README.md` 하나만 유지하고 별도 `*_NOTES.md`는 생성하지 않음

## v2.0.18 변경사항

- 건설 시스템 문제점/편의성 보강
  - 건설 팝업에 선택 후 조작 안내문을 추가해 `건물 선택 -> 팝업 닫힘 -> 바닥 드래그 프리뷰 -> 터치 설치` 흐름을 더 명확하게 표시
  - 건물 선택 직후 stage `pointerdown`에서도 바로 프리뷰를 갱신해, 손가락을 처음 올린 위치부터 초록/빨강 설치 판정이 보이도록 수정
  - 건설 모드 중 우측 `건설` 버튼을 다시 누르면 설치 모드가 취소되도록 변경해 작은 화면에서 취소 동선을 줄임
  - 설치 충돌과 보행 충돌을 더 분리: 꽃밭/분수 같은 소프트 장식은 설치 중복은 막지만 보행 충돌은 과하게 넓히지 않도록 정리
  - 길/건물/장식/바다/기존 시설과 겹치는 설치는 계속 빨강 판정으로 차단

- 퀘스트/메뉴 화면 드래그 스크롤 재보강
  - 메뉴 화면 공통으로 `v2018-menu-drag-screen`을 추가하고 모바일 pointer 기반 세로 드래그 스크롤 보조 로직을 설치
  - 버튼이나 카드 위에서 손가락을 시작해도 일정 거리 이상 움직이면 클릭보다 스크롤을 우선하도록 처리
  - 퀘스트 카드, 추천 의뢰 보드, 전체 의뢰 리스트에 `touch-action: pan-y`를 다시 부여하고 하단 안전 여백을 확장
  - 밝은 패널 위 텍스트에는 계속 어두운 stroke/그림자 처리를 적용해 흰 글씨가 묻히는 문제를 줄임

- 시작 화면 로그인 유지 버튼 추가 축소
  - `이 기기에서 로그인 유지` 버튼 폭을 v2.0.17보다 한 번 더 줄임
  - 익명 서버연동 버튼과의 간격을 더 벌리기 위해 로그인 유지 버튼을 약간 더 아래로 이동
  - 작은 화면에서도 체크 영역과 문구가 눌려 보이지 않도록 indicator와 글자 크기를 함께 재조정

- 우측 조작 UI와 하단 도크 재보정
  - 마을 우측 `+`, `-`, 원점, 건설, 출항 버튼의 높이와 간격을 다시 줄여 세로로 길어 보이는 느낌을 완화
  - 건설 모드 중에는 건설 버튼에 `취소` 보조 라벨이 떠서 현재 상태를 바로 알 수 있게 함
  - 우측 최하단 메뉴 도크 아이콘은 살짝 더 키우되, 큰 테두리/빈칸 버튼/화면 밖 잘림 방지 규칙은 유지

- 캐릭터 방향/움직임 안정화 유지
  - 8방향 파일명은 계속 실제 바라보는 화면 방향 기준으로 고정
  - `west -> east`, `east -> west` 같은 반대 보정표가 다시 들어오면 validate에서 실패
  - 걷기 연출에 미세한 스쿼시/스트레치 보정을 추가하되, 이름표는 계속 `scale(1, 1)`로 고정해 뒤집히지 않게 유지

- 루미나 베이 추가 에셋 적용
  - 기존 보유 에셋 중 `notice_board`, `plaza_stairs`, `bridge_asset`을 새 장식 타입으로 연결
  - 광장 안내판, 작은 계단, 항구 보조 다리 장식을 추가 배치해 항구/광장/해변 밀도를 보강
  - service worker precache에도 신규 사용 에셋을 추가해 오프라인 캐시 누락을 방지

- 버전과 검증 갱신
  - `package.json`, `package-lock.json`, `APP_VERSION`, service worker cache, offline badge를 `2.0.18`로 통일
  - `npm run validate`는 v2.0.18 건설 편의성, 퀘스트 드래그 스크롤, 방향 반전 방지, 신규 에셋 존재, registry 방어, README-only 규칙을 함께 확인
  - 루트 문서는 계속 `README.md` 하나만 유지
  - 별도 `*_NOTES.md`는 생성하지 않음

## v2.0.15 변경사항

- validate-and-deploy 워크플로우 설치 실패 대응
  - `package-lock.json`에 섞여 있던 내부 OpenAI/Artifactory 프록시 resolved URL을 공용 npm registry URL로 교체
  - `applied-caas`, `artifactory`, `internal.api.openai`, `10.192.*` 토큰이 lockfile에 남아 있으면 검증에서 실패하도록 안전장치 추가
  - `.npmrc`를 추가해 CI와 로컬 설치 모두 `https://registry.npmjs.org/`를 기본 registry로 사용
  - `ci:install` 스크립트에 registry 고정과 fetch retry 옵션을 추가

- GitHub Pages validate-and-deploy 안정화
  - `.github/workflows/pages.yml`에서 `npm run ci:install`을 사용하도록 정리
  - 설치 전 npm registry를 공용 npm으로 강제 설정
  - 설치 전 `package-lock.json` 내부 registry 오염 여부를 먼저 검사
  - `pull_request`에서는 validate/build까지만 수행하고 Pages 배포는 push/manual 실행에서만 수행

- 다음 패치 전 필수 점검 항목 추가
  - GitHub Actions의 `validate-and-deploy` workflow run에서 `Install`, `Static validate`, `Typecheck`, `Build`가 모두 통과하는지 확인
  - 실패 시 먼저 `package-lock.json`에 내부 registry URL이 다시 들어갔는지 확인
  - 새 패치 생성 시 `npm run validate`가 v2.0.15 CI registry 검사를 반드시 포함하는지 확인

- 기존 플레이/UX 패치 유지
  - v2.0.13 캐릭터 방향 반전 보정 유지
  - v2.0.14 원화 배경 제거, 메뉴 도크 빈칸 제거, 스크롤 보정 유지
  - 루트 문서는 계속 `README.md` 하나만 유지
  - 별도 `*_NOTES.md`는 생성하지 않음

### validate-and-deploy 확인 순서

1. GitHub 저장소의 Actions 탭으로 이동
2. `validate-and-deploy` workflow 선택
3. 최신 run을 열고 아래 단계 확인
   - `Force public npm registry` 통과
   - `Guard package-lock registry` 통과
   - `Install` 통과
   - `Static validate` 통과
   - `Typecheck` 통과
   - `Build` 통과
4. `Install`에서 `packages.applied-caas` 또는 `10.192.*`가 다시 보이면 lockfile이 오염된 것이므로 v2.0.15 패치를 다시 기준으로 사용

## 조작 방식

- 좌측 조이스틱: 자유 이동
- 마을 타일 터치: 해당 위치로 이동
- NPC 근처 터치: 이동 후 대화창 표시
- 여관, 어시장, 낚시 길드, 항구 사무소 터치: 건물 내부 팝업 표시
- 우측 `+`, `-`: 플레이어 중심 확대/축소
- 우측 `◎`: 플레이어 위치로 카메라 복귀
- 우측 `건설`: 건설 팝업 열기/닫기
- 우측 `출항`: 낚시 화면 이동
- 우측 하단 메뉴: 가방, 퀘스트, 지도, 마을 화면 진입용 2x2 간소 메뉴

## 에셋 반영 원칙

- 새 에셋은 한 번에 무리하게 전부 교체하지 않고, 화면별로 안정적으로 반영합니다.
- 우선순위는 다음과 같습니다.
  1. 캐릭터/NPC 스프라이트 안정화
  2. 대화창 + NPC 초상화
  3. 상단 HUD/우측 하단 메뉴/팝업 UI 프레임
  4. 건물 내부 진입 화면
  5. 월드맵 UI
  6. 낚시 미니게임 UI/VFX
- 이미지 파일명은 기능을 알 수 있게 유지합니다.
- 루트에는 에셋 설명용 별도 MD를 만들지 않습니다. 필요한 설명은 이 README에만 적습니다.
